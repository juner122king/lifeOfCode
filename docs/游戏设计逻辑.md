# 《代码人生》游戏设计逻辑

本文档整理当前版本的系统设计、玩法循环和核心参数关系。事实来源为 `src/game.js`、`src/content.js`、`src/tui.js`、现有测试以及 `docs/人物卡.md`。本文档不逐项复制全部内容表数值；需要查每个内容项的完整字段时，以 `src/content.js` 为准。

如果只需要查看资源、基础属性、技能和项目之间的职责边界与流向关系，见 [资源、属性、技能、项目自洽模型](系统自洽模型.md)。

## 1. 核心定位

《代码人生》是纯文字放置/经营式程序员成长游戏。玩家选择一个人物卡开档，在持续时间结算中安排当前行动，通过活动积累资源、学习技能、购买工具、交付项目、领取目标奖励并晋升职业。

核心循环：

1. 创建或加载档案。
2. 选择初始人物卡，获得差异化属性、资源、技能和少量活动等级。
3. 选择当前推进方向：活动、技能学习或项目。
4. 时间推进时持续结算当前方向。
5. 用产出的资源学习技能、购买工具、启动项目。
6. 项目交付获得金钱、声望、技能经验和属性经验。
7. 目标系统引导阶段性成长，职业晋升提高精力上限并给属性经验。

同一时间只能有一个 active 模式：

- `activeActivityId`：持续活动。
- `activeSkillLearningId`：耗时学习技能。
- `activeProjectId`：推进项目工时。

启动任意一个模式会清空另外两个模式。`stop` 可停止活动、暂停学习或暂停项目。

## 2. 状态模型

新状态由 `createNewState()` 创建，核心字段如下。

| 系统 | 字段 | 作用 |
| --- | --- | --- |
| 档案 | `profileId`、`profileName`、`characterCardId` | 当前存档身份和人物卡来源 |
| 时间 | `worldTimeMinutes`、`lastTick`、`createdAt`、`updatedAt` | 世界日历和现实时间结算 |
| 资源 | `resources` | 代码、金钱、知识、精力、Bug、技术债、压力等 |
| 属性 | `attributes`、`attributeExp`、`attributeBreakthroughs` | 基础能力、属性经验和突破 |
| 职业 | `currentRole` | 决定精力上限、晋升目标和每日预算修正 |
| 活动 | `activityLevels`、`activityExp`、`activityStats`、`activeActivityId` | 活动等级、经验、累计活动秒数和当前活动 |
| 技能 | `skillProgress`、`skillLearningProgress`、`unlockedSkills`、`activeSkillLearningId` | 技能等级/经验、学习进度和当前学习 |
| 项目 | `projectProgress`、`activeProjectId`、`activeProjectDeadlines`、`completedProjects` | 项目投入、工时、Deadline 和完成记录 |
| 收藏 | `ownedTools`、`claimedGoals` | 已拥有工具和已领取目标 |
| 统计 | `stats` | 累计代码、修复 Bug、完成项目数 |

默认属性：

| 属性 id | 名称 | 默认值 | 主要影响 |
| --- | --- | ---: | --- |
| `logic` | 逻辑 | 22 | 降低 Bug/技术债影响，提高质量处理 |
| `focus` | 专注 | 24 | 降低活动精力消耗，常作为技能/活动条件 |
| `learning` | 学习 | 28 | 缩短技能学习耗时 |
| `communication` | 沟通 | 18 | 缓解项目压力风险，影响业务/沟通路线 |
| `resilience` | 抗压 | 20 | 缓解压力惩罚，提高恢复和抗压路线 |
| `creativity` | 创造 | 16 | 影响创意/产品/前端路线 |

默认资源：

- `money` 30。
- `energy` 为当前职业精力上限，实习程序员为 100。
- 其他生产资源和风险资源初始为 0。

资源钳制：

- `bugs`、`techDebt`、`reputation`、`knowledge`、`tests`、`docs`、`architecture`、`leads`、`codeLines`、`money` 不低于 0。
- `pressure` 限制在 0-100。
- `energy` 限制在 0 到当前职业 `maxEnergy`。

## 3. 档案与人物卡

档案系统支持多个角色状态隔离：

- 默认档案 id 为 `default`，名称为 `默认档案`。
- 新建档案必须选择人物卡：`profile new <id> --card <cardId> [name]`。
- 加载档案前会自动保存当前档案。
- 不能删除 `default` 档案，不能删除当前正在使用的档案。

人物卡设计目标是提供强差异开局，而不是线性强弱。当前有 8 张人物卡：

| id | 名称 | 设计定位 |
| --- | --- | --- |
| `academy-prodigy` | 象牙塔学霸 | 高学习/逻辑，适合系统学习；资金、沟通、抗压弱 |
| `indie-hacker` | 野路子独立开发者 | 高创造/专注，自带 JavaScript 和写功能 Lv.2 |
| `seasoned-contractor` | 熟练外包枪手 | 高抗压/专注，资源多但质量风险高 |
| `open-source-zealot` | 开源社区极客 | 高逻辑/创造，自带 Git、文档和声望 |
| `determined-switcher` | 破釜沉舟转行者 | 高沟通/抗压、资金充足，但逻辑/学习弱 |
| `product-minded-dev` | 产品型前端 | 高沟通/创造，自带线索、文档、HTML/CSS |
| `perfectionist-qa` | 完美主义细节控 | 高专注/逻辑，自带测试自动化和测试资产 |
| `laid-back-slacker` | 乐天派“面条哥” | 高抗压、休息 Lv.2，但专注/创造弱且有初始风险 |

人物卡应用逻辑：

- 覆盖初始基础属性。
- 对默认资源做 delta 加减，负数会受资源下限保护。
- 赋予初始技能等级，至少 Lv.1。
- 赋予少量活动等级；当前只有 3 张卡给活动 Lv.2。
- 当前版本所有人物卡都不自带工具。

完整人物卡参数见 `src/content.js` 和 `docs/人物卡.md`。

## 4. 时间、日历与暂停

时间结算由 `settleTime(state, now)` 驱动：

- 使用现实时间差 `now - lastTick`，按秒取整。
- 每 1 秒现实时间推进 1 单位 `worldTimeMinutes`，显示为世界分钟。
- 单次离线结算上限为 8 小时现实时间，即 `OFFLINE_CAP_SECONDS = 28800`。
- 结算按最多 60 单位切段，跨日时触发新一天逻辑。

世界日历：

- 起始时间为 09:00。
- 1 天 1440 分钟。
- 1 周 7 天。
- 1 月 4 周，即 28 天。
- 1 年 12 月，即 336 天。

TUI 自动结算：

- 正常情况下每 3 秒调用一次 `settleTime` 并保存。
- 玩家按 Space 暂停时停止自动结算，并冻结进度条动画。
- 暂停不改变当前 active 模式；恢复后继续原活动、学习或项目。

每日预算：

- 基础为 480m。
- 受职业、精力、压力修正后，落到最近的桶值：360 / 420 / 480 / 540 / 600。
- 职业修正：实习 +0，初级 +60，中级 +120，高级 +60。
- 精力 >= 80 时 +60；精力 < 50 时 -60；精力 < 30 时再 -60。
- 压力 >= 50 时 -60；压力 >= 70 时再 -60。
- 当有 active 工作且当日已用时间达到预算后，进入低效加班。

## 5. 资源、属性与倍率关系

资源分为生产资源、消耗资源和风险资源：

| 类型 | 资源 |
| --- | --- |
| 生产/成长 | `codeLines` 代码、`knowledge` 知识、`tests` 测试、`docs` 文档、`architecture` 架构、`leads` 线索、`reputation` 声望 |
| 经济/体力 | `money` 金钱、`energy` 精力 |
| 风险/负担 | `bugs` Bug、`techDebt` 技术债、`pressure` 压力 |

属性有效值：

- `effectiveAttribute = baseAttribute + breakthrough * 0.2`。
- 当前属性成长只会提高基础属性，突破字段保留但暂无主要玩法入口。

属性经验：

- 每个属性独立累计 `attributeExp`。
- 属性从当前值升 1 点的成本为 `50 + currentAttribute * 5`。
- 基础属性最高 100。

通用属性加成：

- 多数公式使用 `attributeBonus(attr, perPoint, maxBonus)`。
- 只计算超过 20 的有效属性：`min(maxBonus, max(0, effective - 20) * perPoint)`。

技能和工具倍率：

- 已学技能按技能等级指数叠乘：`multiplier *= value ^ level`。
- 已拥有工具按 1 级叠乘。
- 叠乘后有全局上下限：
  - `code` 最高 1.8。
  - `money` 最高 1.6。
  - `bug`、`debt`、`pressure` 最低 0.55。

风险对产出的影响：

- 压力降低代码效率，抗压可缓解，压力惩罚最高 0.35。
- 技术债降低代码效率，债务惩罚最高 0.25。
- 技术债会提高 Bug 风险，逻辑可缓解，最高额外 0.5。

## 6. 活动系统

活动是最基础的持续产出方式。当前有 14 个活动：

`feature-coding`、`bug-hunting`、`refactoring`、`study`、`testing`、`documentation`、`freelancing`、`open-source`、`architecture`、`code-review`、`performance-tuning`、`prompt-engineering`、`incident-response`、`rest`。

活动字段：

- `tier`：活动层级，主要影响内容定位。
- `primaryAttribute`：主要属性，影响活动正向产出和活动经验。
- `requirements`：解锁条件，包括技能、活动等级、职业。
- `effectsPerSecond`：每秒正向或负向资源变化。
- `risksPerSecond`：每秒风险资源变化。
- `energyCostPerSecond`：精力消耗。
- `activityExpPerSecond`：活动等级经验。
- `attributeExpPerMinute`：每分钟给的属性经验。

活动结算关系：

- 活动等级倍率：`1 + (level - 1) * 0.08`。
- 主属性倍率：`1 + attributeBonus(primaryAttribute, 0.0025, 0.22)`。
- 精力倍率：非休息活动按当前精力/上限在 0.35-1 之间缩放。
- 精力低于 30 时，非休息活动再乘 0.8。
- 加班时正向产出乘 0.45。
- 学习周增强 `study`；质量周增强 `bug-hunting`、`refactoring`、`testing`、`code-review`。
- `codeLines` 受代码倍率和生产风险影响。
- `money` 受对应倍率影响。
- `money` 在外包周增强，在质量周削弱。
- `bugs`、`techDebt`、`pressure` 的降低效果分别受逻辑、逻辑、抗压影响。
- 低精力会额外增加压力；精力为 0 时非休息活动收益下降。

活动升级：

- 活动等级从 Lv.1 起步。
- 升级所需经验为 `120 + level * 80`。
- 活动经验由 `activityExpPerSecond * seconds` 叠加，并受主属性倍率、加班、周重点影响。

内容平衡层：

- `src/content.js` 中活动先写 raw 参数，再统一缩放。
- 正向资源产出、负向质量改善、风险增长都有独立缩放表。
- 活动经验统一乘 0.45。

## 7. 技能系统

当前有 30 个技能，分为 1-5 阶。技能提供倍率、项目门槛和晋升门槛。

技能学习：

- 未学技能可执行 `learn <skillId>`。
- 学习前检查属性要求。
- 第一次学习需要支付 `cost`，之后暂停/继续不重复付费。
- 学习会设置 `activeSkillLearningId`，并清空当前活动和项目。
- 学习耗时为 `learningSeconds * (1 - learningRelief)`。
- `learningRelief = attributeBonus(learning, 0.0025, 0.2)`，最多减少 20% 学习耗时。
- 学习完成后技能等级至少变为 Lv.1，清空学习进度，并同步 `unlockedSkills`。

技能默认成本：

| tier | 默认资源成本 | 默认学习耗时 |
| --- | --- | ---: |
| 1 | knowledge 120 / money 80 | 600 |
| 2 | knowledge 300 / money 220 | 1200 |
| 3 | knowledge 650 / money 520 | 2100 |
| 4 | knowledge 1100 / money 900 | 3300 |
| 5 | knowledge 1700 / money 1600 | 5400 |

部分技能覆盖默认成本。实际配置以 `src/content.js` 为准。

技能等级：

- 等级名：未学、入门、熟练、精通、专家、大师。
- 升级技能需要技能经验、资源和更高属性要求。
- 升级经验阈值：Lv.1 -> 2 需要 120，Lv.2 -> 3 需要 320，Lv.3 -> 4 需要 700，Lv.4 -> 5 需要 1250。
- 升级属性要求为基础学习属性要求 + `6 * (targetLevel - 1)`。
- 升级知识成本取 `max(最低知识, 学习知识成本 * 倍率)`。
- 升级额外资源来自 `upgradeResourceBase`，乘以目标等级倍率。

升级知识最低值：

| 目标等级 | 最低 knowledge |
| --- | ---: |
| Lv.2 | 120 |
| Lv.3 | 280 |
| Lv.4 | 520 |
| Lv.5 | 900 |

升级资源倍率：

| 目标等级 | knowledge 倍率 | 其他资源倍率 |
| --- | ---: | ---: |
| Lv.2 | 2 | 1 |
| Lv.3 | 4 | 2 |
| Lv.4 | 7 | 4 |
| Lv.5 | 11 | 7 |

## 8. 工具系统

当前有 8 个工具：

`used-laptop`、`keyboard`、`jetbrains`、`ai-assistant`、`api-client`、`cloud-ide`、`github-actions`、`status-page`。

工具逻辑：

- 通过 `buy <toolId>` 购买。
- 只检查金钱或配置中的资源成本。
- 购买后加入 `ownedTools`，不能重复购买。
- 工具倍率直接参与全局 `getMultipliers`。

工具定位：

- 工具是中后期的稳定倍率来源。
- 与技能不同，工具没有等级，不消耗学习时间。
- 工具可作为目标条件，例如拥有工具数量。

## 9. 项目系统

当前有 46 个项目，包括 11 个主项目、大量训练项目和高级综合项目。

项目字段：

- `difficulty`：1-5 难度。
- `requirements.resources`：启动时一次性投入的资源。
- `requirements.skills`：要求技能；检查时所需技能等级等于项目难度，最低 1，最高 5。
- `requirements.activityLevels`：要求活动等级。
- `minWorkHours`：基础工时，内容构建时会按难度放大。
- `maxSuccessRate`：最高成功率。
- `rewards`：交付奖励。
- `skillExpRewards`：项目技能经验奖励。
- `attributeExp`：首次成功时给的属性经验。

内容构建层：

- 如果项目未手写资源/奖励，会使用难度模板。
- 启动资源按难度缩放：难度 1/2/3/4/5 分别为 1.6 / 2 / 2.4 / 3 / 3.5。
- 工时按难度缩放：难度 1/2/3/4/5 分别为 1.5 / 1.75 / 2 / 2.25 / 2.5。
- 奖励中的金钱统一乘 0.75 后向上取整。
- 训练项目统一为难度 1，主要用于给指定技能经验。

项目启动：

- 执行 `project <projectId>`。
- 如果已有投入，则再次启动不重复扣资源。
- 启动后设置 `activeProjectId`，清空当前活动和学习。
- 若没有 Deadline，会创建 Deadline：使用项目配置的 `deadlineDay`，否则按当前日 + `max(2, ceil(deadlineDays || 3 + difficulty * 2))`，到期时间为当天 18:00。

项目推进：

- 每段结算增加 `workedSeconds * workMultiplier`。
- 项目周提高项目推进；学习周降低项目推进。
- 精力 < 30 时项目推进乘 0.8。
- Deadline 逾期后项目推进乘 0.85，项目奖励乘 0.75，并成功率额外 -0.12。

成功率：

- `riskScore = bugRisk + debtRisk + pressureRisk`。
- Bug 风险：`clamp(bugs / 100, 0, 1) * 0.4`。
- 技术债风险：`clamp(techDebt / 180, 0, 1) * 0.35`。
- 压力风险：`clamp(pressure / 100, 0, 1) * 0.25 * (1 - communicationRelief)`。
- `communicationRelief = attributeBonus(communication, 0.003, 0.22)`。
- `successRate = maxSuccessRate - riskScore * difficulty * 0.12`。
- 最终成功率限制在 0.15 到 `maxSuccessRate`。

交付结果：

- 工时达标后掷随机数决定成功或失败。
- 成功时：
  - 首次成功给完整金钱、声望、属性经验和完成计数。
  - 重复成功只给 5% 金钱奖励，不重复给声望、属性经验和完成计数。
  - 给项目配置的技能经验，受世界事件技能经验倍率影响。
  - 清空项目进度和 Deadline。
- 失败时：
  - 已投入资源损失。
  - 给 40% 技能经验。
  - 清空项目进度。

Deadline：

- 逾期首次检查会标记 warned，压力 +10，并提示成功率/奖励受惩罚。
- 逾期超过 2 天宽限期后项目失败，清空进度，声望 -2，压力 +18。

## 10. 目标、晋升和路线引导

当前有 17 个目标，分主线和支线。

目标系统：

- 目标可设置前置目标 `requiresGoals`。
- 目标要求可检查统计、活动统计、活动等级、资源、技能、完成项目、工具数量和职业。
- 状态为：未解锁、进行中、可领取、已领取。
- 领取目标给金钱、声望和属性经验。
- `claim all` 会循环领取所有当前可领取目标，直到没有可领取目标。

主线引导：

- 当前主线优先展示第一个未领取且前置满足的主线目标。
- 如果没有前置满足的主线，则展示第一个未领取主线。
- 全部主线完成后显示主线已完成。

职业系统：

当前职业：

| id | 名称 | 精力上限 | 晋升到 |
| --- | --- | ---: | --- |
| `intern` | 实习程序员 | 100 | junior |
| `junior` | 初级程序员 | 120 | middle |
| `middle` | 中级程序员 | 140 | senior |
| `senior` | 高级程序员 | 160 | 无 |

晋升要求：

- 实习 -> 初级：声望 2、完成项目 1、HTML/CSS、JavaScript、写功能 Lv.3、系统学习 Lv.2。
- 初级 -> 中级：声望 8、完成项目 3、Git、SQL、写测试 Lv.4、写文档 Lv.3、重构 Lv.4。
- 中级 -> 高级：声望 18、完成项目 5、Docker、沟通能力、架构 Lv.6、开源协作 Lv.5、接外包 Lv.4。

晋升成功：

- 切换职业。
- 精力恢复 20，但不超过新职业上限。
- 获得新职业配置的属性经验。

## 11. 周重点与世界事件

周重点通过 `week <id>` 设置，当前可选：

| id | 名称 | 效果 |
| --- | --- | --- |
| `learning` | 学习周 | `study` 和技能学习 1.3，项目推进 0.8 |
| `project` | 项目周 | 项目推进 1.3，技能学习 0.8 |
| `freelance` | 外包周 | 金钱产出 1.3，压力风险 1.2 |
| `quality` | 质量周 | 质量活动 1.3，代码和金钱产出 0.85 |
| `balanced` | 均衡周 | 无偏置 |

世界事件由世界日历触发和生效：

| id | 名称 | 时间 | 效果 |
| --- | --- | --- | --- |
| `ai-boom` | AI 热潮 | D29-D84 | LLM Agent 学习与项目技能经验 x2，项目奖励 x1.1 |
| `hiring-freeze` | 招聘冻结 | D113-D140 | 项目奖励 x0.8，触发时压力 +5 |
| `open-source-season` | 开源季 | D169-D196 | 触发时声望 +1 |

随机事件：

- 仅在 `randomEvents: true` 的结算中触发。
- 触发概率为 `min(0.35, seconds / 3600 * 0.12)`。
- 事件从 13 个随机事件中抽取，直接修改资源或给予属性经验。

## 14. 设计约束与维护注意

- 游戏设计的主要平衡点在 `src/content.js` 内容表和缩放函数。
- 结算、状态归一化和命令行为集中在 `src/game.js`。
- TUI 只消费 option/view model，不应复制核心结算逻辑。
- 新增内容时优先补测试，现有测试覆盖存档、活动、技能、项目、目标、人物卡、档案和 TUI helper。
- 修改资源、技能倍率或项目成功率时，需要同时检查早期目标、晋升门槛和项目门槛，避免卡死成长线。
