# 每小时资源汇总功能设计

**日期**: 2026-06-16  
**状态**: ✅ 已实现

## 概述

在游戏日志中添加每小时资源和进展汇总功能，让玩家能够清晰地了解每个小时的收益和变化。

## 需求

- **汇总间隔**: 每游戏小时（60 分钟游戏时间）
- **触发时机**: 严格按整点触发（10:00、11:00、12:00...），但跳过 00:00（该时刻有日终总结）
- **汇总内容**: 完整报告，包括资源变化、活动进展、属性经验、技能等级提升等所有变化
- **显示方式**: 特殊类型的多行日志，有独特的 [汇总] 标签和颜色

## 数据结构

### State 新增字段

```javascript
state.lastHourlySummaryHour = null; // 上次汇总的小时数（0-23）
state.hourlySummarySnapshot = {
  resources: {},      // 资源快照 {energy: 100, pressure: 20, ...}
  activityLevels: {}, // 活动等级快照 {activityId: {level, exp}}
  attributeExp: {},   // 属性经验快照 {logic: 50, creativity: 30, ...}
  worldMinute: 0      // 快照时的游戏分钟数
};
```

### 初始化

- **新档案创建时**: `lastHourlySummaryHour` 设为当前小时（避免立即触发）
- **快照初始化**: 复制当前状态的资源、活动等级、属性经验

## 触发逻辑

### 检测整点

在 `settleTime` 主循环的每个时间段推进后：

1. 获取当前游戏时间的小时数（`Math.floor((state.worldTimeMinutes % 1440) / 60)`）
2. 检查条件：
   - 当前小时 ≠ `lastHourlySummaryHour`
   - 当前小时 ≠ 0（跳过 00:00，因为有日终总结）
   - `lastHourlySummaryHour` ≠ null（避免游戏启动时立即触发）
3. 如果满足条件，触发每小时汇总

### 特殊情况处理

**跨天处理:**
- 23:xx → 00:xx 时不触发汇总（00:00 有日终总结）
- 00:xx → 01:00 正常触发

**离线结算:**
- 如果离线时间跨越多个整点，只在当前整点触发一次汇总
- 汇总内容覆盖整个离线期间的变化

**优先级:**
- 日终总结 > 阶段总结 > 每小时汇总
- 如果同一时刻触发多个总结，跳过每小时汇总

## 汇总内容生成

### `generateHourlySummary(state)` 函数

对比当前状态与 `hourlySummarySnapshot`，生成完整报告。

#### 1. 时间范围

格式：`HH:00-HH:00`（例如："10:00-11:00"）

从快照时间到当前时间。

#### 2. 资源变化

对比所有资源（精力、压力、Bug、技术债、金钱、声望、知识）：

```
精力 -15（65→50），压力 +10（20→30），Bug -2（15→13）
```

- 只显示有变化的资源
- 格式：`资源名 变化量（前值→后值）`
- 如果所有资源无变化，显示："无明显变化"

#### 3. 活动进展

记录这一小时的主要活动：

```
活动：写功能 60分钟，Lv.2 → Lv.3
```

- 显示活动名称和时长
- 如果等级有变化，显示等级提升
- 如果有多个活动，列出主要的（时长最长的）

#### 4. 属性经验

对比属性经验总量：

```
属性：逻辑 +12，韧性 +8
```

- 只显示有增长的属性
- 格式：`属性名 +增长量`

#### 5. 其他重要事件

- 技能学习完成
- 项目阶段完成
- 活动等级提升（除了已在活动进展中显示的）

### 输出格式

多行文本，每个维度一行：

```
[汇总] 10:00-11:00
资源：精力 -15（65→50），压力 +10（20→30），Bug -2（15→13）
活动：写功能 60分钟，Lv.2 → Lv.3
属性：逻辑 +12，韧性 +8
```

### 快照更新

生成汇总后，立即更新快照：
- 复制当前资源、活动等级、属性经验
- 更新 `worldMinute` 为当前时间
- 更新 `lastHourlySummaryHour` 为当前小时

## 日志显示

### 事件类型

- **新增类别**: `"hourly_summary"`
- **标签**: `[汇总]`
- **严重程度**: `"info"`

### TUI 集成

在 `src/tui.js` 中：

1. **`INFO_EVENT_LABELS` 添加**:
   ```javascript
   hourly_summary: "汇总"
   ```

2. **颜色配置**:
   - 建议使用 `THEME.status.info` 或 `THEME.title`
   - 与普通事件有明显视觉区分

3. **多行显示**:
   - 多行文本在事件日志中正常换行
   - 每行缩进保持一致

### 可见性

- 汇总日志出现在玩家信息窗口的事件日志区域
- 与其他事件（项目完成、技能提升等）混合显示
- 按时间顺序排列

## 实现要点

### 文件修改

2. 在 `settleTime` 主循环中添加整点检测
3. 实现 `generateHourlySummary` 函数
4. 实现 `updateHourlySummarySnapshot` 函数

**`src/tui.js`:**
1. 在 `INFO_EVENT_LABELS` 中添加 `hourly_summary: "汇总"`
2. 为 `hourly_summary` 类别配置显示颜色

### 错误处理

- 如果快照不存在，使用当前状态作为基准（变化量为 0）
- 如果资源数据缺失，跳过该资源的显示
- 多行文本格式化时确保换行符正确

### 测试场景

1. **正常整点触发**: 从 09:50 推进到 10:10，应在 10:00 触发汇总
2. **跨天边界**: 从 23:50 推进到 00:10，不应触发汇总
3. **离线结算**: 离线 3 小时，应只触发一次汇总
4. **首次启动**: 游戏启动时不应立即触发汇总
5. **阶段转换冲突**: 12:00 同时是阶段结束和整点，应只显示阶段总结

## 决策记录

- **为什么选择整点触发？** 便于玩家记忆和预期，与现实时间习惯一致
- **为什么跳过 00:00？** 避免与日终总结重复，保持日志清晰
- **为什么使用快照而不是累积？** 简化实现，降低存储开销，与阶段总结逻辑一致
- **为什么只在最后一个整点触发离线汇总？** 避免日志过载，保持可读性
