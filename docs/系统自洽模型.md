# 资源、属性、技能、项目自洽模型

本文档定义《代码人生》中“资源、基础属性、技能、项目”的职责边界和流向关系。它不是数值平衡表，也不修改当前实现；具体字段和数值仍以 `src/content.js`、`src/game.js` 为准。

## 1. 一句话模型

核心循环：

```text
人物卡 -> 属性/资源/初始技能 -> 活动产出资源 -> 资源+属性学习技能 -> 技能+资源启动项目 -> 项目奖励反哺技能/属性/资源
```

这条循环里，每类概念只承担一部分职责：

- 资源：短期状态、库存、项目材料和风险水位。
- 基础属性：长期能力底盘，主要负责倍率、门槛、风险缓解和成长方向。
- 技能：可学习的专业能力，负责倍率、项目门槛、晋升门槛和项目经验承接。
- 项目：资源消耗池、技能验证场和阶段性成长结算点。

设计目标是避免把资源、属性、技能、项目都写成同一种“会涨的数值”。资源可以频繁涨跌，属性慢速积累，技能通过学习和项目验证成长，项目负责把阶段投入结算成新的成长空间。

## 2. 当前实现范围

本文档描述当前版本已经存在的关系：

- 人物卡会覆盖初始属性，并给资源 delta、初始技能等级、少量活动等级；当前人物卡不自带工具。
- 活动是持续产出和风险变化的主要入口，也会给活动经验和属性经验。
- 学习技能需要属性门槛、资源成本和学习时间；学习期间会暂停活动和项目。
- 技能升级需要技能经验、资源成本和更高属性门槛。
- 项目启动检查资源、活动等级和技能等级；项目资源只在首次启动或重新启动未投入项目时扣除一次。
- 项目推进到工时后按成功率结算；首次成功给金钱、声望、完成记录、技能经验和属性经验，重复成功只给少量金钱并继续给技能经验。
- 项目失败会清空项目进度和已投入资源，并给 40% 项目技能经验。
- 目标会给资源或属性经验，晋升会给属性经验和职业变化，但它们不是本文档的主角。

本文档也给出后续维护时建议遵守的职责边界。凡是“建议”“不建议”“应该”开头的内容是设计原则，不表示当前代码已经强制校验。

## 3. 四类概念边界

| 概念 | 核心问题 | 当前主要字段 | 应该承担 | 不应该承担 |
| --- | --- | --- | --- | --- |
| 资源 | 现在手上有什么、风险水位多高 | `resources` | 活动产物、项目材料、学习成本、购买成本、风险状态 | 长期人格能力、专业掌握程度 |
| 基础属性 | 这个人擅长什么、短板在哪里 | `attributes`、`attributeExp` | 学习门槛、学习耗时缓解、活动倍率、风险缓解、路线倾向 | 直接替代技能、直接作为大额货币 |
| 技能 | 已经掌握哪些专业能力 | `skillProgress`、`skillLearningProgress` | 产出/风险倍率、项目门槛、晋升门槛、项目技能经验承接 | 频繁消耗型库存、短期压力/债务水位 |
| 项目 | 用已有能力交付什么成果 | `projectProgress`、`completedProjects`、`activeProjectDeadlines` | 消耗资源、验证技能、结算奖励、推进主线 | 稳定日常产出来源、无门槛提款机 |

## 4. 资源职责

资源是最短周期的状态层。它们可以被活动快速产出或压低，也可以被学习、工具和项目消费。

当前资源可以分为三组：

| 类型 | 资源 | 主要含义 |
| --- | --- | --- |
| 生产/成长资源 | `codeLines` 代码、`knowledge` 知识、`tests` 测试、`docs` 文档、`architecture` 架构、`leads` 线索、`reputation` 声望 | 可积累的工作产物或外部认可 |
| 经济/体力资源 | `money` 金钱、`energy` 精力 | 购买、学习、行动效率和作息惩罚的基础 |
| 风险/负担资源 | `bugs` Bug、`techDebt` 技术债、`pressure` 压力 | 降低产出效率、影响项目成功率或触发惩罚的风险水位 |

资源的职责：

- 作为活动的直接产出：例如写功能产出代码，系统学习产出知识，写测试产出测试资产。
- 作为学习和升级的成本：技能学习消耗知识、金钱等；技能升级继续消耗资源。
- 作为项目启动材料：项目一次性消耗代码、测试、文档、架构、线索等材料。
- 作为风险水位：Bug、技术债、压力影响代码效率和项目成功率。
- 作为奖励承载：项目、目标、随机事件可以增加金钱、声望等资源。

资源不应该承担的职责：

- 不应该长期决定角色路线。路线差异应优先来自人物卡属性、技能选择、项目选择和活动等级。
- 不应该单独解锁高阶项目。高阶项目应同时看技能、活动等级和材料库存。
- 不应该替代技能经验。项目给技能经验，是为了让“做过对应项目”反哺专业能力。

## 5. 基础属性职责

基础属性是人物长期能力底盘。当前属性包括：

| 属性 id | 名称 | 当前主要作用 |
| --- | --- | --- |
| `logic` | 逻辑 | 提高逻辑类活动表现，缓解技术债导致的 Bug 风险，常见于后端、质量、架构路线 |
| `focus` | 专注 | 缓解活动精力消耗，影响持续产出和前端/测试类学习门槛 |
| `learning` | 学习 | 缩短技能学习耗时，影响系统学习路线和高阶技能进入 |
| `communication` | 沟通 | 缓解项目压力风险，影响外包、文档、开源和业务沟通路线 |
| `resilience` | 抗压 | 缓解压力惩罚，提高恢复、救火和抗压路线表现 |
| `creativity` | 创造 | 影响创意、产品、前端、AI 相关路线 |

属性的职责：

- 作为人物卡差异的底层表达：人物卡首先改变的是属性底盘，而不是只给资源。
- 作为技能学习门槛：技能配置中的 `attributeRequirements` 决定是否能学习。
- 作为技能升级门槛基础：升级时属性要求会随目标等级提高。
- 作为活动倍率来源：活动的 `primaryAttribute` 会提高活动产出和活动经验。
- 作为风险缓解来源：例如沟通缓解项目压力风险，逻辑缓解技术债带来的 Bug 风险。
- 作为作息和加班缓解来源：抗压提高恢复和压力缓解，学习提高技术浏览知识收益，专注缓解加班折损和副业次日精力惩罚，创造/沟通分别提高副业金钱/声望收益。
- 作为长期奖励：活动、项目、目标、晋升、部分随机事件给属性经验，属性经验再转化成基础属性点。

属性不应该承担的职责：

- 不应该直接替代技能门槛。高逻辑不等于已经掌握 SQL 或 Docker。
- 不应该成为频繁消费品。属性是底盘，不是项目材料。
- 不应该大幅直接产出资源。属性应偏倍率和缓解，而不是成为另一个资源生产按钮。

## 6. 技能职责

技能是可学习的专业能力。当前技能有等级，等级名为未学、入门、熟练、精通、专家、大师。

技能的职责：

- 作为专业路线选择：学习 HTML/CSS、JavaScript、Git、SQL、Docker、LLM Agent 等，代表角色实际掌握了某类能力。
- 作为全局倍率来源：技能按等级叠乘 `code`、`money`、`bug`、`debt`、`pressure` 等倍率。
- 作为项目门槛：项目要求的技能需要达到项目难度对应等级，最低 1，最高 5。
- 作为晋升门槛：职业晋升会要求特定技能已经学会。
- 作为项目经验承接对象：项目成功给完整技能经验，失败给 40% 技能经验，重复交付也会继续给技能经验。

技能学习和升级的当前流向：

```text
资源库存 + 属性门槛 -> 学习时间 -> 技能 Lv.1
技能 Lv.n + 技能经验 + 资源库存 + 更高属性门槛 -> 技能 Lv.n+1
```

技能不应该承担的职责：

- 不应该像资源一样被项目直接消耗掉。技能是能力，不是材料。
- 不应该完全替代活动等级。项目可以同时需要技能和活动等级，以区分“会这门技术”和“做过足够多这种工作”。
- 不应该只靠学习闭门升级。项目技能经验让“交付验证”成为技能成长的一部分。

## 7. 项目职责

项目是阶段性成长结算点。它把资源库存、技能等级、活动等级和风险水位综合到一次交付结果里。

项目的职责：

- 消耗资源：启动时扣除 `requirements.resources`，表示把代码、测试、文档、架构、线索等投入到交付里。
- 验证技能：项目要求的每个技能都需要达到项目难度对应等级。
- 验证工作经验：项目可要求活动等级，例如写功能、测试、文档、架构等。
- 承担风险：Bug、技术债、压力会降低项目成功率；沟通可以缓解压力风险。
- 结算成长：成功后给金钱、声望、技能经验和属性经验；首次成功还记录完成项目数。
- 打开后续路线：完成项目可满足目标、晋升或更高阶路线的前置条件。

项目不应该承担的职责：

- 不应该变成纯资源兑换。项目必须看技能、活动经验和风险。
- 不应该成为无限刷属性入口。当前实现中重复成功不重复给声望、属性经验和完成计数，这是合理边界。
- 不应该完全无视失败收益。当前失败仍给 40% 技能经验，说明失败也是专业经验，但不会返还资源。

## 8. 作息基调职责

作息基调是持续生效的生活习惯，而不是日程槽。玩家通过 `lifestyle [id]` 设置明日基调，实际切换在次日 09:00 发生，避免午休或睡前临时切换带来的微操。

作息的职责：

- 在休整窗口提供自动收益或代价：中午、晚上 `none` 和深夜分别按当前基调结算。
- 表达长期生活路线：健康恢复、技术浏览、赛博娱乐、独立副业各自偏向不同资源和风险。
- 承接属性差异：`resilience`、`learning`、`focus`、`creativity`、`communication` 分别影响对应作息收益或减益。
- 让加班有明确机会成本：晚上加班会失去 18:00-21:00 的作息收益，但不额外叠隐藏惩罚。

作息不应该承担的职责：

- 不应该给活动经验或属性经验。尤其 `side_hustle` 只产出少量金钱、极少声望、压力和次日精力惩罚。
- 不应该被当天即时切换套利。当前作息持续到次日 09:00，待切换作息只存在 `pendingLifestyleStanceId`。
- 不应该替代项目和技能路线。副业可以缓解现金流，但不会直接绕开技能、项目材料或风险管理。

## 10. 门槛与奖励分工

为了保持系统自洽，门槛和奖励应按职责分配。

| 用途 | 优先使用 | 谨慎使用 | 避免使用 |
| --- | --- | --- | --- |
| 学习技能门槛 | 基础属性、资源成本 | 前置技能 | 项目完成数、大量活动等级 |
| 升级技能门槛 | 技能经验、基础属性、资源成本 | 特定项目奖励经验 | 纯金钱购买等级 |
| 项目启动门槛 | 资源、技能等级、活动等级 | 职业、已完成项目 | 单一资源数值 |
| 项目成功率 | Bug、技术债、压力、沟通缓解 | Deadline、世界事件 | 纯随机且无可控因素 |
| 活动解锁门槛 | 活动等级、技能、职业 | 属性 | 金钱 |
| 晋升门槛 | 声望、项目数、技能、活动等级 | 特定目标 | 单一属性或单一资源 |
| 奖励资源 | 活动、项目、目标、事件 | 重复项目低额奖励 | 属性直接兑换资源 |
| 奖励属性 | 活动积累、首次项目成功、目标、晋升 | 随机事件少量奖励 | 重复项目无限刷属性 |
| 奖励技能经验 | 项目成功/失败 | 特定活动少量补充 | 普通资源直接购买大量技能经验 |

## 11. 当前核心闭环

### 11.1 开局闭环

```text
选择人物卡
  -> 获得不同属性底盘、资源库存、初始技能或活动等级
  -> 根据短板选择活动
```

人物卡不是“强弱模板”，而是不同风险和路线的起点。比如高学习角色更容易进入技能学习，高沟通角色更适合外包和项目风险缓解，高专注角色适合持续产出。

### 11.2 活动到技能

```text
活动产出 knowledge / money
  -> 属性满足技能要求
  -> 支付技能学习成本
  -> 学习时间完成后得到技能 Lv.1
```

这里资源解决“有没有条件去学”，属性解决“是否适合学”，学习时间解决“需要投入多久”。三者缺一，会让技能学习变成单一货币购买。

### 11.3 技能到项目

```text
活动产出项目材料
  -> 技能达到项目难度对应等级
  -> 活动等级达到项目要求
  -> 启动项目并投入资源
```

项目同时检查材料、技能和活动等级，是为了区分三种能力：

- 材料：手上有可投入的代码、测试、文档、架构等资产。
- 技能：知道怎么做。
- 活动等级：实际做过足够多类似工作。

### 11.4 项目到成长

```text
项目工时达标
  -> 根据风险水位计算成功率
  -> 成功：资源奖励 + 技能经验 + 首次属性经验/声望/完成记录
  -> 失败：投入损失 + 部分技能经验
```

项目奖励同时连接资源、技能和属性，但不把三者混成同一种奖励：

- 金钱和声望是外部回报。
- 技能经验是专业实践反馈。
- 属性经验是首次里程碑带来的长期成长。

## 12. 风险水位的定位

Bug、技术债、压力属于资源字段，但它们不是普通库存，而是风险水位。

当前风险影响：

- Bug、技术债、压力共同降低项目成功率。
- 技术债会降低代码效率，并提高 Bug 风险。
- 压力会降低代码效率；抗压可缓解压力惩罚。
- 沟通可缓解项目压力风险。
- 某些活动能降低 Bug、技术债或压力，某些活动会增加它们。

风险水位的设计意义：

- 让“高产出活动”存在长期代价。
- 让质量活动有明确价值，而不是只给低收益资源。
- 让项目不是单纯工时条，而是一次风险管理检查。
- 让属性短板能通过风险形式表现出来。

建议保持风险资源的“可管理但不可忽视”：

- 风险过低时不应阻塞玩法。
- 风险过高时应明显影响项目成功率和产出效率。
- 降低风险的活动应产出较少直接收益，但提供项目稳定性。

## 13. 后续维护原则

新增资源时，先回答它属于哪一类：

- 生产材料：是否会被项目消耗？
- 成长货币：是否会被学习或升级消耗？
- 风险水位：是否会影响效率、成功率或惩罚？
- 外部认可：是否会进入晋升、目标或项目奖励？

新增属性时，先回答它和现有六项是否真的不同。属性应影响门槛、倍率、风险缓解或路线选择，不能只是“另一种经验条”。

新增技能时，至少明确三件事：

- 学习它需要哪些属性和资源。
- 它提供什么倍率或风险修正。
- 它服务哪些项目、晋升或路线。

新增项目时，至少明确四件事：

- 它消耗哪些资源材料。
- 它验证哪些技能，并对应什么难度。
- 它要求哪些活动等级来表示实践经验。
- 它首次成功奖励哪些技能经验和属性经验。

调数值时，不要只看单点收益。需要同时检查：

- 活动产出是否能支撑技能学习成本。
- 技能学习和升级是否能支撑项目门槛。
- 项目奖励是否能反哺下一阶段学习、工具或项目。
- 风险水位是否让质量活动和恢复活动有实际意义。
- 晋升门槛是否仍能通过主线项目和常规活动自然达到。
